# Постбеки

> **Примечание о gate'е регистрации.** Регистрационный gate в боте (Screen 1b) — **не единственный** способ закрыть доступ. Дополнение — **парольный gate** (Screen 1c), настраиваемый через поле `bots.access_password`. Пароль не отключает постбек-gate: если заданы оба поля (`referral_url_template` и `access_password`), оба варианта предлагаются параллельно на одном экране — пользователь сам выбирает, что удобнее (зарегистрироваться по партнёрской ссылке ИЛИ ввести код доступа). Gate проходит, если сработал любой из двух путей. Подробности — см. [bot-flow.md → Screen 1c: Password Gate](bot-flow.md#screen-1c-password-gate).

## Макросы: наши ↔ партнёрские

Интеграция состоит из двух направлений:

1. **Исходящее** — наша реферальная ссылка, которую бот показывает пользователю. В неё мы подставляем свои значения в партнёрские capture-поля.
2. **Входящее** — партнёрский postback, в параметрах которого партнёр возвращает нам те же значения через свои макросы.

### Таблица 1 — наши плейсхолдеры в `bots.referral_url_template`

Подстановка делается в [bot/handlers/start.py `_build_referral_url`](../bot/handlers/start.py) перед показом кнопки «Зарегистрироваться».

| Наш плейсхолдер | Подставляется | Куда кладём (имя query-параметра) |
|-----------------|---------------|-----------------------------------|
| `{user_id}` | `telegram_id` пользователя | `click_id` |
| `{bot_id}` | ID нашего бота | `sub_id1` |

**Канонический шаблон ссылки (сохраняется в админке):**

```
https://<partner>/smart/...?click_id={user_id}&sub_id1={bot_id}
```

> **Важно:** `site_id` не работает — партнёр перезаписывает его своим внутренним значением (всегда `0`). Поэтому bot_id передаём через `sub_id1`.

### Как работает возврат данных

Партнёр **не подставляет** макросы `{click_id}`, `{sub_id1}` внутри URL постбека. Вместо этого он **автоматически дописывает** все capture-поля (`click_id=...`, `sub_id1=...`, `site_id=...`) как дополнительные query-параметры к заданному URL постбека. Поэтому в URL постбека достаточно указать только `secret` и `status`/`event` — остальное придёт автоматически.

### Таблица 2 — параметры нашего postback-эндпоинта

Это то, что партнёр присылает нам в `GET /postback?...`. В кабинете партнёра используется их макрос, который раскроется в конкретное значение при отправке постбека.

Партнёр автоматически дописывает к URL постбека следующие параметры:

| Параметр (авто) | Наш маппинг | Описание |
|-----------------|-------------|----------|
| `click_id` | `telegram_id` | Telegram ID пользователя (из `click_id` в tracking-ссылке) |
| `sub_id1` | `bot_id` | ID нашего бота (из `sub_id1` в tracking-ссылке) |
| `site_id` | _(игнорируется)_ | Внутренний ID партнёра, всегда `0` — **не использовать** |

Явные параметры (задаём в URL постбека):

| Параметр | Описание |
|----------|----------|
| `event` (или `status`) | Фиксированное значение: `reg`, `ftd`, `redep`, `commission`, `withdrawal` |
| `secret` | Значение `bots.postback_secret`. НЕ логируется. |

**Приоритет чтения в эндпоинте:**
- telegram_id: `click_id` → `user_id` (первое непустое)
- bot_id: `sub_id1` → `bot_id` → `site_id` (первое непустое ненулевое)
- event: `event` → `status` (первое непустое)

## Канонический URL для партнёра

Поскольку партнёр автоматически дописывает `click_id` и `sub_id1` к URL, в самом URL достаточно указать только `secret` и `event`:

```
https://<base_url>/postback?secret=<postback_secret>&event=reg
```

- `<base_url>` — значение из `settings.postback_base_url` (см. migration 017).
- `click_id=<telegram_id>` и `sub_id1=<bot_id>` — **дописываются партнёром автоматически**.
- `event=...` — фиксированное значение; для каждого события партнёр настраивает отдельный постбек.
- `<postback_secret>` — значение из `bots.postback_secret` конкретного бота.

Для каждого события партнёр настраивает отдельный постбек, меняя только `event=...`.

**Пример (бот 2, секрет 777):**

```
https://mforexbot.yooriko.com/postback?secret=777&event=reg
```

Партнёр автоматически добавит: `&click_id=<tg_id>&sub_id1=2&site_id=0&...`

## События (event)

| `event` | Описание | Когда приходит |
|---------|----------|----------------|
| `reg` | Регистрация трейдера | Трейдер зарегистрировался на платформе партнёра |
| `ftd` | First Time Deposit | Первый депозит трейдера — **pass-criterion для депозитного гейта (Screen 1d)** в дополнение к обновлению `users.status` |
| `redep` | Повторный депозит | Любой депозит после первого |
| `commission` | Комиссионное начисление | Ежедневно, за каждого активного трейдера |
| `withdrawal` | Вывод средств | Статус — в `params_json.status`: `new`, `processed`, `canceled` |

`reg` используется для gate'а регистрации (Screen 1b), `ftd` — для депозитного гейта (Screen 1d). Остальные события просто логируются в `postback_events` и могут быть использованы для будущей логики (аналитика, уведомления).

### Обновление `users.status` при постбеке

При успешном постбеке (`auth_status='ok'`) PHP обновляет поле `users.status`:

| `event` | Новый `status` | Условие |
|---------|----------------|---------|
| `reg` | `registered` | Только если текущий статус = `new` |
| `ftd` | `deposited` | Только если текущий статус != `deposited` |

Статус **только повышается**, никогда не понижается. Подробнее о жизненном цикле — см. [bot-flow.md → User Status Lifecycle](bot-flow.md#user-status-lifecycle).

FTD также открывает депозитный гейт (Screen 1d) — та же строка `postback_events` с `event_type='ftd'` и `auth_status='ok'` является pass-criterion для SELECT, который выполняет бот при клике «Я сделал депозит». См. [bot-flow.md → Screen 1d: Deposit Gate](bot-flow.md#screen-1d-deposit-gate).

### Уведомление пользователю при регистрации

При `event=reg` с `auth_status='ok'`, PHP отправляет пользователю сообщение в Telegram. Текст берётся из ключа перевода `postback.reg_congrats` (язык = `users.lang_code`). Содержит плейсхолдер `{support_link}`, подставляемый из `bots.support_link`. Отправка — fire-and-forget через Bot API (`sendMessage`), ошибки логируются, но не влияют на HTTP-ответ постбека.

**Важно:** при `event=ftd` **никакое DM не отправляется** — это осознанное продуктовое решение. Пользователь увидит успешный переход к главному меню в самом боте, когда повторно нажмёт кнопку «✅ Я сделал депозит» на экране депозитного гейта (Screen 1d). Никакого ключа `postback.ftd_congrats` не существует и не должен добавляться.

### Уведомление в админ-группу

При **любом** постбеке с `auth_status='ok'`, если у бота задан `bots.admin_group_id` (Telegram chat ID группы администраторов), PHP отправляет уведомление в эту группу.

**Содержание сообщения:** имя бота, telegram_id пользователя, username, тип события.

Отправка — fire-and-forget через Bot API (`sendMessage`). Поле `admin_group_id` управляется в Admin -> Боты -> Редактировать (label: "ID группы — админы"). См. [bot-flow.md → Admin: Bot Edit Form](bot-flow.md#admin-bot-edit-form).

## URL постбек-эндпоинта (настройки админки)

Наш приёмный эндпоинт: `GET {POSTBACK_BASE_URL}/postback?bot_id={site_id}&user_id={sub_id1}&event=reg&secret={bots.postback_secret}`

- `POSTBACK_BASE_URL` — глобальная настройка, хранится в таблице `settings` (ключ `postback_base_url`, см. migration 017). Редактируется в админке на `/admin/settings`. Значение шарится между всеми ботами (таблица `settings` **не имеет** `bot_id`).
- Готовый URL для вставки в партнёрский кабинет отображается в `Admin → Боты → Редактировать` под полем Postback Secret. Кнопка «📋 Копировать» копирует URL в буфер. Если `postback_base_url` пуст или у бота не задан `postback_secret`, вместо URL показывается подсказка.
- Admin-форма правит `bots.postback_secret` и `bots.access_password` (парольный gate, см. Screen 1c в bot-flow.md).

## Хранилище: `postback_events`

**Append-only** таблица — каждый входящий постбек записывается отдельной строкой. Переходов статусов и ENUM на `users` нет: «зарегистрирован ли пользователь» определяется наличием строки с нужным `event_type` в `postback_events`.

```sql
postback_events (
    id, bot_id, telegram_id, event_type, auth_status,
    params_json, raw_query, ip, received_at
)
```

- `raw_query` — оригинальная query string «как есть» (с `secret=REDACTED`). Нужен для отладки, когда `params_json` мог потерять нюансы экранирования.
- `auth_status` — `ok | bad_secret | unknown_bot | missing_params`. **Все** входящие запросы логируются (в т.ч. с неверным секретом или неизвестным bot_id), чтобы можно было отлаживать интеграцию и отслеживать попытки перебора.
- `bot_id` и `event_type` допускают NULL, чтобы можно было записать невалидные запросы (отсутствующие параметры, неизвестный бот).

**Важно:** gate в боте считает только строки с `auth_status = 'ok'` — неавторизованные запросы логируются, но не открывают доступ.

Любые дополнительные параметры (`trader_id`, `sumdep`, `commission`, `wdr_sum`, `country`, `device_type`, ...) сохраняются целиком в `params_json` — нет необходимости заранее описывать схему под каждое событие.

## Проверка регистрации в боте (gate)

Бот проверяет факт регистрации простым `SELECT`:

```sql
SELECT 1 FROM postback_events
 WHERE bot_id = ? AND telegram_id = ? AND event_type = 'reg'
 LIMIT 1
```

Если строка найдена → пользователь прошёл gate; иначе показывается экран «Зарегистрируйтесь» с реферальной ссылкой и кнопкой «Я зарегистрировался» (повторная проверка того же SELECT).

## Коды ответа

HTTP-код отражает результат валидации, но строка в `postback_events` пишется **всегда**:

| HTTP | `auth_status` | Условие |
|------|---------------|---------|
| 200 | `ok` | Всё валидно, постбек принят |
| 400 | `missing_params` | Нет `bot_id` (или legacy `site_id`) или `event` |
| 404 | `unknown_bot` | `bot_id` не соответствует записи в `bots` (bot_id в лог пишется как NULL) |
| 401 | `bad_secret` | Неверный `secret` (`hash_equals`) |

Если `user_id` (или legacy `click_id`) не числовой или отсутствует — строка всё равно пишется, но `telegram_id = NULL` (полезно для постбеков, где пользователь не идентифицируется, напр. `commission` по трейдеру).

## Админ-панель: Postback Logs

Страница `/admin/postbacks` — просмотр лога всех входящих постбеков из таблицы `postback_events`.

### Фильтры (верхняя панель)

| Фильтр | Элемент | Описание |
|--------|---------|----------|
| `bot_id` | dropdown | Выбор бота (все / конкретный) |
| `event_type` | text input | Поиск по типу события (reg, ftd, redep, ...) |
| `auth_status` | dropdown | ok / bad_secret / unknown_bot / missing_params |
| `q` | text input | Поиск по `telegram_id` |

### Колонки таблицы

ID, Bot (id + name), Telegram ID, Event, Auth Status, IP, Received At, Params (JSON, первые 50 символов, раскрывается по клику).

### Пагинация

50 записей на страницу, серверная пагинация. Сортировка: `received_at DESC` (новые сверху).

### Цветовая маркировка auth_status

| Статус | Цвет |
|--------|------|
| `ok` | Зелёный фон |
| `bad_secret` | Красный фон |
| `unknown_bot`, `missing_params` | Жёлтый/оранжевый фон |

## Справка: макросы партнёрской платформы

Полный набор макросов, которые партнёр может использовать в настройках постбеков (мы используем только `{site_id}` и `{sub_id1}`, остальные — опциональные параметры, которые улетают в `params_json`):

| Placeholder | Параметр |
|-------------|----------|
| `{site_id}` | ID сайта/источника в вашем трекере |
| `{cid}` | ID кампании в системе Pocket Partners |
| `{ac}` | Название кампании в системе Pocket Partners |
| `{sub_id1}`–`{sub_id5}` | Свободные макросы для любых переменных |
| `{trader_id}` | ID трейдера |
| `{country}` | Страна трейдера |
| `{device_type}` | Устройство трейдера (мобильный / десктоп / планшет) |
| `{os_version}` | ОС трейдера (iOS, Android, Windows, MacOS и т.д.) |
| `{browser}` | Браузер трейдера |
| `{promo}` | Промокод, использованный трейдером |
| `{link_type}` | Лендинг Pocket Partners |
| `{date_time}` | Дата и время события |
| `{sumdep}` | Сумма депозита в USD |
| `{wdr_sum}` | Сумма вывода в USD |
| `{status}` | Статус вывода (в процессе, обработанный, отклонённый) |
| `{commission}` | Сумма комиссии за каждого трейдера |
